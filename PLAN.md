# Founding Plan

## Non-Negotiables

- Do not collapse `Shell`, `Mind`, `Memory`, and `World` into one runtime blob.
- Do not let portrait art become a single hardcoded ASCII file.
- Do not hide continuity only inside transcript text.
- Do not introduce fake terminal flavor with commands that do not map to real subsystems.
- Do not overbuild the first slice into a framework before the product's emotional core is proven.
- Never generate raw ASCII first. Generate invisible world logic first, then render. The renderer is the last pass, not the first. See `WORLD_GENERATION.md` and `WORLDGEN.md` for the world-generation doctrine.

## Design Priorities

1. Prove recognizable presence.
2. Prove continuity and correction.
3. Prove visible evolution.
4. Prove replaceable infrastructure.
5. Productize only after identity and memory feel durable.

## Operating Rules

- All user-facing rituals must map to actual subsystems.
- Every new subsystem must declare its layer ownership.
- Every new memory type must declare retention, visibility, and editability.
- Every visual mutation must declare what caused it and how it can be reversed or preserved.
- Every major repo change should add one short note to `WORKLOG.md`.

## Lean Start

The MVP should load one portrait pack, one theme, and two module packs:

- `portrait-pack.core.sacred-machine`
- `module-pack.core.rituals`
- `module-pack.archive.ledger`

That is enough to prove shell presence, memory surfacing, and identity continuity without inventing a full marketplace too early.

## Immediate Execution Order

1. Implement the shell frame and render contract.
2. Implement slot composition for a minimal portrait.
3. Implement mode state and command routing.
4. Implement session persistence plus one durable profile record.
5. Implement one memory promotion path that creates one visible artifact.
6. Add a controlled dream/reflection path only after the above feels stable.

## Documentation Standard

- Keep docs decisive.
- Keep manifests machine-readable.
- Keep notes short and timestamped.
- Prefer examples that can be directly loaded by future adapters.
